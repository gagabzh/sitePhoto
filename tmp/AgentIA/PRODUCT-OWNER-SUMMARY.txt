📊 RÉSUMÉ DE VOTRE AGENT ACTUEL
═══════════════════════════════════════════════════════════════════════════

✅ POINTS FORTS (Excellent travail!)

  • Format de story très précis (CODE-N, As a [role], action, so [value])
  • Nommage cohérent et prévisible (DS-, US-P, ALB-, etc.)
  • Structure de sections claire et bien organisée
  • Règles strictes pour rejeter les mauvaises stories
  • Acceptance criteria bien compris
  • V5 stories déjà livrées et documentées

⚠️  DOMAINES À AMÉLIORER (Pour intégration QA Agent)

  1. Pas de communication avec les autres agents
     → Comment PO parle avec QA? Aujourd'hui: pas de process.

  2. Pas de framework d'acceptance criteria unifié
     → Chaque story peut avoir des criteria inconsistentes.

  3. Pas de checklist de testabilité
     → QA doit deviner si une story est testable.

  4. Pas de gestion des blockers
     → Blockers découverts trop tard (pendant dev ou QA).

  5. Pas de coordination avec QA avant finalisation
     → Stories finalisées, puis QA découvre des problèmes.

═══════════════════════════════════════════════════════════════════════════

🎯 LA SOLUTION: 5 AMÉLIORATIONS CLÉS
═══════════════════════════════════════════════════════════════════════════

1. ✅ ACCEPTANCE CRITERIA FRAMEWORK
   
   Avant: "Acceptance criteria on besoin" (vague)
   Après: Checklist stricte pour TOUTES les stories
   
   Checklist:
   □ Action est testable (pas "feels smooth")
   □ Edge cases couverts (0, 1, max, invalid)
   □ Error states définis (network down, permission denied)
   □ Performance scopée (si relevant: < 500ms)
   □ Accessibility considérée (keyboard, screen reader)
   □ Mobile behavior défini (si différent du desktop)
   □ Sécurité/accès explicite (qui peut/ne peut pas)
   
   Résultat: 0 ambiguité pour QA


2. ✅ QA COORDINATION WORKFLOW
   
   Avant: PO écrit → Dev code → QA teste (tard!)
   Après: PO écrit → QA valide → Dev code → QA teste
   
   Nouvelle étape:
   1. PO écrit draft story
   2. PO demande QA: "Is this testable?"
   3. QA donne feedback (edge cases, clarifications)
   4. PO revise story
   5. QA sign-off: "Ready to develop"
   6. Story finalisée
   
   Résultat: Blockers trouvés tôt, QA n'est pas surprise


3. ✅ BLOCKERS UPFRONT
   
   Avant: Blockers découverts pendant développement
   Après: Blockers identifiés dans la story
   
   Types:
   • Tech: API pas prête, DB migration, dépendance manquante
   • Product: Conflit de priorité, dépend d'une autre story
   • QA: Besoin d'env spécial, données de test
   
   Résultat: Pas de surprises, planning plus juste


4. ✅ "WHAT QA NEEDS FROM PO" GUIDE
   
   Avant: PO ne sait pas ce que QA trouve flou
   Après: Guide explicite de ce que QA valide
   
   Questions que PO doit répondre:
   • Qui, quoi, quand, pourquoi?
   • Limites de données? (max items, max size)
   • Performance importante? (spécifier max temps)
   • Accès contrôlé? (qui peut/ne peut pas)
   • Erreurs possibles? (network, invalid input)
   
   Résultat: PO écrit des stories QA-ready dès le départ


5. ✅ COORDINATION AVEC AUTRES AGENTS
   
   Avant: PO, QA, Dev, Tech Lead travaillent en silos
   Après: Rôles et interactions explicites
   
   PO ↔ QA: QA valide testabilité avant finalisation
   PO ↔ Planner: PO produit stories, Planner les planifie
   PO ↔ Dev: Dev demande clarifications si critères ambigus
   PO ↔ Tech Lead: Tech Lead valide architecture + sécurité
   
   Résultat: Workflow clair, moins d'aller-retours

📋 EXEMPLE: AVANT vs APRÈS
═══════════════════════════════════════════════════════════════════════════

❌ AVANT (Story incomplète, ambigu):

    US-V5 — Browse photos in album
    As a viewer, I can scroll through an album's photos,
    so I can see the collection.

    Acceptance Criteria:
    - Photos are displayed
    - User can navigate
    - Works on mobile

    → QA se pose:
    ✗ Scroll ou pagination?
    ✗ Si 0 photos?
    ✗ Si 10,000 photos?
    ✗ Quel est le load time acceptable?
    ✗ Keyboard navigation?
    ✗ Quel rôle peut faire ça?

    → QA demande de clarifications
    → Dev attend
    → Timeline glisse


✅ APRÈS (Story complète, testable):

    US-V5 — Browse photos in album by page
    As a viewer, I can scroll through an album's photos one page at a time,
    so I can see the collection without loading everything at once.

    Acceptance Criteria:
    - Each page shows 20 photos with Previous/Next buttons
    - Next disabled on last page, Previous disabled on page 1
    - If album empty, show "No photos in this album"
    - Each page loads < 500ms (show spinner if slower)
    - Mobile: 48×48px minimum tap targets
    - Keyboard nav: Tab through buttons, Enter navigates
    - Only viewers of shared albums can see photos

    QA Notes:
    □ Test with albums: 0, 1, 20, 21, 1000 photos
    □ Test network delay (spinner visible?)
    □ Test keyboard + screen reader

    → QA immédiatement: "Testable, ready"
    → Dev peut commencer
    → QA sait exactement comment tester
