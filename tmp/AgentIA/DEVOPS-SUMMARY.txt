🎯 LES 7 AMÉLIORATIONS CLÉS
═══════════════════════════════════════════════════════════════════════════

1. ✅ INFRASTRUCTURE TASK READINESS CHECKLIST

   Avant: DevOps touches prod, assume task is ready
   Après: Validate 10 points before proceeding

   Checklist:
   □ Task description clear? (what, why, expected outcome)
   □ Impact analyzed? (services, users, capacity)
   □ Dependencies clear? (what blocks this)
   □ Rollback plan documented? (detailed, tested)
   □ Testing strategy defined? (health checks, metrics, smoke tests)
   □ Blockers identified? (quotas, timing, dependencies)
   □ Deployment window? (off-peak, maintenance needed)
   □ Monitoring plan? (alerts, dashboards, thresholds)
   □ Scaling implications? (capacity, performance, cost)
   □ Security impact? (firewall, SSH, encryption)

   Si pas OK → Ask Planner avant proceeding


2. ✅ DEFINITION OF DONE (Infra-Specific)

   Avant: "Done" = implicite, subjective
   Après: Explicit checklist

   5 Sections:
   • Planning & Documentation (rollback, impact, testing)
   • Implementation (staging first, validation, secrets)
   • Production Deployment (backup, monitoring, notification)
   • Post-Deployment Validation (health, performance, users)
   • Documentation & Knowledge Transfer (runbooks, lessons)


3. ✅ ROLLBACK PLAN DOCUMENTATION

   Avant: "Hope it works", no documented revert plan
   Après: For every change, detailed rollback steps

   Must include:
   • What changed and why
   • Specific step-by-step rollback procedure
   • Conditions for rollback (when to execute)
   • Estimated rollback time
   • Pre-requisites (tested, team trained)


4. ✅ INFRASTRUCTURE VALIDATION CHECKLIST

   Avant: Manual spot checks, inconsistent
   Après: Systematic validation approach

   4 Categories:
   • Service Health (containers, DB, Redis, worker)
   • Performance Baseline (CPU, memory, disk, network)
   • Application Smoke Tests (login, upload, display, search)
   • Monitoring Active (dashboards, alerts, logs)


5. ✅ BLOCKER COMMUNICATION

   Avant: Ad-hoc, if DevOps mentions it
   Après: Tracked, escalated to Planner

   Format:
   🔴 BLOCKER: [Code]
     Description: [What's blocked]
     Type: [Hardware/API/Dependency]
     Severity: [Critical/High/Medium]
     Impact: [Which tasks blocked]
     Status: [OPEN/INVESTIGATING]
     Owner: [DevOps/OVH/Other]
     Resolve by: [Date]


6. ✅ CLEAR PR SCOPE BOUNDARIES

   Avant: Can mix infra + code in one PR
   Après: Separated (clear boundaries, clear ownership)

   DevOps PRs: terraform/, docker-compose, .github/workflows/
   Dev PRs: src/, worker/src/, public/, migrations/

   Rule: NEVER mix in one PR


7. ✅ COORDINATION WITH DEVELOPER

   Avant: "Just deploy" without coordination
   Après: "Infrastructure ready?" protocol

   Process:
   Dev asks: "Infrastructure ready?"
   DevOps confirms: "Infrastructure validated"
   Dev deploys app (knowing infra is stable)
