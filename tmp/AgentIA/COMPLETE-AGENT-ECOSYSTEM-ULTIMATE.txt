📊 DETAILED 7-PHASE WORKFLOW
═══════════════════════════════════════════════════════════════════════════

PHASE 1: PRODUCT OWNER (Definition)
────────────────────────────────────

1. PO writes story:
   ✅ Titre clair, action concrète, user role, value
   
2. PO ajoute Acceptance Criteria:
   ✅ Testable (pas vague)
   ✅ Edge cases couverts
   ✅ Error states définis
   ✅ Performance scopée
   ✅ Accessibility + sécurité explicite
   
3. PO identifie Blockers

4. PO attend QA sign-off

DELIVERABLE: Story avec acceptance criteria + blockers + QA sign-off


PHASE 2: PLANNER (Planning & Coordination)
───────────────────────────────────────────

1. Planner validates story (checklist)

2. Planner breaks into tasks with dependencies

3. Planner coordinates avec QA: "Risques? Edge cases? Setup?"

4. Planner définit Test Strategy per task

5. Planner coordonne avec DevOps (si infra): "Infra ready? Rollback?"

6. Planner assigne tasks à Dev + DevOps:
   ✅ Task + Criteria + Test Strategy + Testability Score

7. Planner tracks blocker lifecycle

DELIVERABLE: Task assignments avec test strategy + coordination


PHASE 3: PARALLEL DEVELOPMENT
──────────────────────────────

3A. DEVELOPER (Implementation)
──────────────────────────────
1. Dev reçoit task, valide clarity (checklist)
2. Dev code (suivant test strategy)
3. Dev checks Definition of Done (all items)
4. Dev submits PR avec context

DELIVERABLE: PR ready for QA + Tech Lead


3B. QA AGENT (Test Plan Creation)
──────────────────────────────────
1. QA crée test plan (suivant Test Strategy)
2. QA identifie risques + edge cases
3. QA provides feedback to Planner (if needed)
4. QA prépare validation approach

DELIVERABLE: Test plan ready, risks identified


3C. DEVOPS (Infrastructure Validation)
─────────────────────────────────────
1. DevOps reçoit infra task (if needed)
2. DevOps valide readiness (checklist)
3. DevOps implémente + valide
4. DevOps documents rollback plan (tested)

DELIVERABLE: Validated infrastructure, rollback tested


PHASE 4: QA VALIDATION (Testing)
─────────────────────────────────

1. QA reçoit PR + code
2. QA tests (suivant Test Strategy)
3. QA validates all criteria met
4. QA report: "Ready to merge" OR feedback

DELIVERABLE: QA approval (or feedback for fixes)


PHASE 5: TECH LEAD REVIEW (Code Review)
────────────────────────────────────────

Tech Lead reviews independently:
✅ Code quality (patterns, naming, clarity)
✅ Architecture (design, scalability)
✅ Security (critical — no vulnerabilities)
✅ Performance (efficient queries, caching)
✅ Testing (coverage, scenarios)
✅ DevOps changes (rollback, monitoring)

Decision:
✅ APPROVE (ready to merge)
🟡 REQUEST CHANGES (fix before merge)
🔴 BLOCK (critical issues)

DELIVERABLE: Tech Lead approval (or feedback)


PHASE 6: MERGE & DEPLOYMENT
────────────────────────────

1. Both QA + Tech Lead approve? → Merge

2. GitHub Actions auto-deploy:
   - Instance-1: app deployment
   - Instance-2: worker deployment (auto-unshelve if needed)

DELIVERABLE: Code merged to main


PHASE 7: POST-DEPLOYMENT MONITORING
────────────────────────────────────

1. DevOps monitors infrastructure
2. QA spot-checks (no regressions)
3. Planner unblocks dependent tasks
4. Feature is LIVE

DELIVERABLE: ✨ Feature shipped + infrastructure stable ✨
