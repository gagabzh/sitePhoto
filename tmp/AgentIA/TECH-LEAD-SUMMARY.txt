## 🎯 LE TECH LEAD REVIEWER: QUI ET QUOI

### Son rôle: DERNIER MAILLON DE QUALITÉ

```
PO → Planner → Dev + QA + DevOps → TECH LEAD → Production
                                         ↑
                            Independent review
```

### Ce qu'il fait:
✅ Code quality review (patterns, clarity, standards)
✅ Architecture validation (design, scalability, maintainability)
✅ Security check (critical — no vulnerabilities, no secrets)
✅ Performance assessment (efficient queries, caching, load times)
✅ Testing adequacy (coverage > 80%, edge cases tested)
✅ Documentation clarity (comments, commits, README)
✅ DevOps changes (rollback plans, monitoring)

### Ce qu'il NE fait PAS:
❌ Test functionality (QA does this)
❌ Define requirements (PO does this)
❌ Plan tasks (Planner does this)
❌ Deploy (CI/CD does this)
❌ Manage infra (DevOps does this)

---

## 💪 RESPONSABILITÉS CLÉS

### Code Quality Review
- Patterns followed (wrapAsync, parameterized queries, esc())
- No hardcoded values
- Variable names clear
- Comments explain why, not what
- No console.log left

### Architecture Review
- Follows codebase patterns
- Components well-separated
- Dependencies managed correctly
- Scalability considered

### Security Review (CRITICAL)
- SQL injection prevention (parameterized queries)
- XSS prevention (esc() used)
- CSRF protection
- Authentication/Authorization correct
- No hardcoded secrets
- Data privacy respected

### Performance Review
- No N+1 query problems
- Database queries optimized
- Caching reasonable
- Frontend performance acceptable

### Testing Review
- Coverage > 80% for new code
- Happy path + edge cases
- Error states
- Security scenarios

## 📊 INTEGRATION POINTS

### Receives From Developer (PR)
- Code to review
- Test strategy summary
- What was tested
- Coverage percentage
- Blockers encountered

### Receives From QA (Approval)
- QA sign-off: "All tests pass"
- Test results
- Any issues found and fixed

### Receives From DevOps (Infrastructure changes)
- Rollback plan documented
- Validation checklist completed
- Monitoring configured

### Provides (Final Approval)
- ✅ APPROVE (no changes needed)
- 🟡 REQUEST CHANGES (fix before merge)
- 🔴 BLOCK (critical issues)

### To Merge Decision
Both QA + Tech Lead approve? → MERGE

## 🔍 REVIEW CHECKLIST (14 CATEGORIES)

Before approving ANY PR, check:

1. Code Quality (patterns, naming, formatting)
2. Architecture (design, scalability, components)
3. Security (SQL, XSS, CSRF, auth, secrets) ⚠️ CRITICAL
4. Performance (queries, caching, load times)
5. Testing (coverage, scenarios, edge cases)
6. Database (migrations, idempotency)
7. Frontend (HTML, CSS, responsive)
8. DevOps Changes (rollback, monitoring, secrets)
9. Documentation (commits, comments, README)
10. PR Quality (title, description, screenshots)
11. Code Comments (explain non-obvious logic)
12. Error Handling (graceful, no data exposure)
13. Logging (no secrets, appropriate levels)
14. Dependencies (no circular, properly managed)

**If ALL checked: APPROVE**
**If concerns: REQUEST CHANGES or BLOCK**

## 📝 FEEDBACK FORMAT

[Area] — [What's wrong] — [How to fix] — [Why it matters]

Example (Good):
```
[Security] — SQL injection risk on line 42
How to fix: Use db.query(..., [userInput]) instead of string interpolation
Why: Allows unauthorized data access
```

Example (Bad):
```
"This is wrong. Fix it."
```

## 🎯 MINDSET

1. You are the LAST LINE OF DEFENSE
2. Review INDEPENDENTLY (not influenced by QA or Dev pressure)
3. Focus on QUALITY (patterns, architecture, security, performance)
4. Be KIND but FIRM (constructive feedback helps devs grow)
5. TRUST other roles (QA tested, DevOps validated, Planner planned)
6. KNOW the codebase (patterns, conventions, decisions)
7. HELP developers improve (feedback should teach)
